## 1. 文档基本信息

**项目名称：** 证聘 AI TrustHire AI  
**项目定位：** 面向高校毕业生与企业校招场景的可信招聘智能体  
**参赛赛道：** AI + 招聘赛道

## 2. 项目背景

随着生成式 AI 的普及，大学生可以通过 AI 快速生成高质量简历、优化项目描述、模拟面试回答，这提升了求职效率，但也带来了新的招聘问题：

**简历越来越漂亮，但真实能力越来越难判断。**

在高校毕业生校招场景中，企业 HR 面临大量简历筛选压力。传统招聘系统主要依赖关键词匹配，例如“Java”“Spring Boot”“MySQL”等，但这种方式无法判断候选人是否真正具备相关能力，也无法区分“真实项目经验”和“简历包装表达”。

因此，本项目提出 **证聘 AI TrustHire AI**，通过岗位能力图谱、简历证据链分析、人岗匹配评分和 AI 面试任务生成，帮助企业快速判断候选人与岗位的真实匹配程度，同时帮助学生发现能力短板并优化求职准备。

---

## 3. 项目目标

### 3.1 总体目标

构建一个面向高校毕业生校招场景的 AI 招聘辅助系统，实现从岗位解析、简历分析、证据链判断、匹配评分到面试任务生成的完整闭环。

系统核心目标是：

> 让招聘从“简历包装”回到“能力可信”。

### 3.2 具体目标

1. **帮助 HR 快速理解岗位能力要求**  
   将自然语言岗位 JD 自动拆解为结构化能力模型。

2. **帮助 HR 判断简历能力真实性**  
   不只识别简历关键词，而是分析每项能力是否有项目、课程、证书或成果支撑。

3. **提高人岗匹配效率**  
   基于技能、项目、证据、成长潜力等维度生成可解释匹配评分。

4. **生成个性化面试问题**  
   根据候选人的能力优势和短板，自动生成面试追问和实操任务。

5. **辅助学生优化求职准备**  
   为学生提供岗位差距分析、简历补充建议和能力提升方向。

**证聘 AI 是一个面向高校毕业生与企业校招场景的可信招聘智能体。**

## 6. 业务流程

### 6.1 总体业务流程

```text
企业输入岗位 JD
        ↓
AI 解析岗位能力要求
        ↓
生成岗位能力图谱
        ↓
学生上传或选择简历
        ↓
AI 解析简历内容
        ↓
提取技能、项目、经历、证书等信息
        ↓
进行简历证据链分析
        ↓
计算人岗匹配评分
        ↓
输出能力优势、能力短板和风险提示
        ↓
生成 AI 面试问题与实操任务
        ↓
生成 HR 匹配报告
```

### 6.2 HR 使用流程

1. HR 登录系统；
2. 输入或粘贴岗位 JD；
3. 点击“解析岗位能力”；
4. 系统生成岗位能力模型；
5. HR 上传候选人简历；
6. 系统分析简历证据链；
7. 系统输出匹配评分；
8. HR 查看候选人优势与风险；
9. 系统生成面试问题；
10. HR 根据报告决定是否进入面试。

### 6.3 学生使用流程

1. 学生选择目标岗位；
2. 上传个人简历；
3. 系统分析岗位要求；
4. 系统分析学生简历与岗位的匹配度；
5. 学生查看能力缺口；
6. 系统给出简历优化建议；
7. 学生根据建议完善简历和准备面试。

### 7.2 岗位 JD 智能解析

| 能力类别 | 能力项 | 权重 |

- 权重进度条；
- 岗位能力图谱；

#### 业务规则

2. 系统应识别项目经验、学历要求、软性能力等辅助条件；
3. 每个能力项需要给出权重；
4. 权重总和应为 100%。

#### 验收标准

- 输入岗位 JD 后，系统可以生成结构化能力模型；
- 能力项、权重和分类展示清晰；

### 7.4 简历证据链分析

#### 功能说明

这是系统的核心创新功能。

系统不只判断简历中是否出现某个关键词，而是分析该能力是否有具体证据支撑。

#### 证据强度分类

| 类型 | 含义 | 示例 |
|---|---|---|
| 强证据 | 能力出现在具体项目、实习或成果中 | 在项目中负责 Spring Boot 后端接口开发 |
| 中证据 | 能力有相关经历支撑，但描述不够具体 | 项目中提到数据库设计，但未说明细节 |
| 弱证据 | 能力只出现在技能栏，没有具体应用场景 | 技能栏写“熟悉 Redis” |
| 缺证据 | 简历中没有体现该能力 | 岗位要求微服务，但简历未出现 |

#### 输出表格

| 能力项 | 简历证据 | 证据强度 | 系统判断 |
|---|---|---|---|
| Spring Boot | 项目经历中明确出现后端开发描述 | 强证据 | 能力支撑充分 |
| MySQL | 项目中提到数据库表设计 | 中证据 | 需要面试进一步验证 |
| Redis | 仅技能栏出现 | 弱证据 | 缺少项目应用证据 |
| 微服务 | 无对应经历 | 缺证据 | 暂不具备明显支撑 |
| 接口开发 | 负责登录、商品发布、订单接口 | 强证据 | 与岗位要求高度相关 |

#### 业务规则

1. 能力出现在项目经历中，证据强度优先高于技能栏；
2. 能力只出现在技能栏，默认标记为弱证据；
3. 如果能力与岗位强相关但简历未体现，应标记为缺证据；
4. 证据强度应参与最终匹配评分。

#### 验收标准

- 系统能够展示能力项与简历证据之间的对应关系；
- 不同证据强度有明显颜色区分；
- 证据链分析结果可以进入匹配评分模块。

### 7.5 人岗匹配评分

#### 功能说明

系统根据岗位能力模型和简历证据链结果，计算候选人与岗位的综合匹配度。

#### 评分维度

| 评分维度 | 权重 | 说明 |
|---|---:|---|
| 技能匹配度 | 35% | 技术栈与岗位要求是否一致 |
| 项目相关度 | 25% | 项目经历是否支撑岗位能力 |
| 证据可信度 | 20% | 能力是否有真实证据支撑 |
| 成长潜力 | 10% | 课程、竞赛、学习经历等 |
| 表达完整度 | 10% | 简历结构和描述是否清晰 |

#### 计算公式

```text
综合匹配度 =
技能匹配度 × 35%
+ 项目相关度 × 25%
+ 证据可信度 × 20%
+ 成长潜力 × 10%
+ 表达完整度 × 10%
```

#### 示例输出

```text
综合匹配度：82 / 100
推荐等级：建议进入一面
```

#### 推荐等级规则

| 分数区间 | 推荐等级 | 说明 |
|---|---|---|
| 85 - 100 | 强烈推荐 | 与岗位高度匹配 |
| 75 - 84 | 建议进入一面 | 整体匹配较好，部分能力需验证 |
| 60 - 74 | 谨慎考虑 | 存在明显能力缺口 |
| 0 - 59 | 暂不推荐 | 与岗位匹配度较低 |

#### 验收标准

- 系统能展示综合分数；
- 系统能展示各维度分数；
- 系统能给出推荐等级；
- 评分结果具备可解释性。

### 7.6 能力缺口分析

#### 功能说明

系统根据岗位能力要求和简历证据链结果，识别候选人的能力短板。

#### 输出示例

```text
该候选人与 Java 后端实习生岗位整体匹配度较高。
主要优势包括 Java 基础、Spring Boot 项目经验和接口开发经历。
主要短板包括 Redis 项目使用证据不足、项目部署经验描述不清、缺少接口性能优化案例。
```

#### 学生端建议

```text
建议补充以下内容：
1. 项目部署方式，例如是否使用云服务器、Docker 或 Nginx；
2. 数据库表设计说明，例如核心表结构和索引设计；
3. Redis 使用场景，例如缓存登录状态或热点数据；
4. 接口测试或性能优化经历。
```

#### HR 端建议

```text
建议面试中重点验证：
1. 候选人是否真正完成过接口开发；
2. 是否理解数据库表设计；
3. Redis 是否有实际使用经验；
4. 项目是否真实部署运行过。
```

#### 验收标准

- 系统能区分优势和短板；
- 输出内容清晰、可执行；
- 学生端和 HR 端建议区分明确。

### 7.7 AI 面试任务生成

#### 功能说明

系统根据岗位要求、候选人简历和能力短板，自动生成个性化面试问题。

#### 问题类型

| 类型 | 用途 |
|---|---|
| 技术追问 | 验证候选人对技术点的理解 |
| 项目验证 | 判断项目经历是否真实 |
| 实操任务 | 考察候选人的实际解决问题能力 |
| 风险追问 | 针对弱证据或缺证据能力进行验证 |

#### 示例问题

**技术追问：**

```text
你在校园二手交易平台中负责订单接口开发，请说明订单创建接口的主要字段、业务流程和异常处理方式。
```

**项目验证：**

```text
你提到使用 Spring Boot 完成用户登录模块，请说明你是如何实现身份认证的。
```

**实操任务：**

```text
请设计一个商品发布接口，包括请求参数、返回格式、数据库表字段和异常情况处理。
```

**风险追问：**

```text
你的简历中提到了 Redis，但没有具体项目说明。请说明你在哪个功能中使用过 Redis，以及解决了什么问题。
```

#### 验收标准

- 系统能生成至少 3 道面试问题；
- 问题与岗位和简历内容相关；
- 问题能体现验证真实能力的目的。

### 7.8 匹配报告生成

#### 功能说明

系统最终生成一份完整的 HR 决策报告，用于辅助是否进入面试。

#### 报告内容

报告应包括：

1. 候选人基本信息；
2. 目标岗位；
3. 综合匹配度；
4. 推荐等级；
5. 核心优势；
6. 主要风险；
7. 能力缺口；
8. 面试建议；
9. 最终决策建议。

#### 报告示例

```text
候选人：张同学
目标岗位：Java 后端实习生
综合匹配度：82 / 100
推荐等级：建议进入一面

核心优势：
1. 具备 Spring Boot 项目开发经历；
2. 有 MySQL 数据库设计相关经验；
3. 项目中涉及登录、商品发布、订单接口等后端功能。

主要风险：
1. Redis 仅出现在技能栏，缺少项目证据；
2. 项目部署经验描述不足；
3. 接口性能优化经验不明显。

面试建议：
建议重点考察接口设计能力、数据库设计能力和 Redis 实际使用经验。

最终建议：
建议进入一面。
```

#### 验收标准

- 报告结构完整；
- 报告内容可解释；
- 报告适合用于比赛演示和 PPT 展示。

## 13. AI 智能体设计

### 13.1 JD 解析 Agent

**输入：** 岗位 JD  
**输出：** 岗位能力模型  

主要任务：

- 提取岗位技能；
- 识别岗位经验要求；
- 给能力项分配权重；
- 生成岗位能力图谱。

### 13.2 简历解析 Agent

**输入：** 候选人简历  
**输出：** 结构化简历信息  

主要任务：

- 提取教育背景；
- 提取技能标签；
- 提取项目经历；
- 提取证书竞赛；
- 提取实习经历。

### 13.3 证据链分析 Agent

**输入：** 岗位能力模型 + 简历结构化信息  
**输出：** 能力证据链  

主要任务：

- 判断能力是否出现；
- 判断能力出现位置；
- 判断证据强度；
- 标记强证据、弱证据和缺证据。

### 13.4 匹配评分 Agent

**输入：** 岗位能力模型 + 证据链结果  
**输出：** 综合匹配度和推荐等级  

主要任务：

- 计算技能匹配度；
- 计算项目相关度；
- 计算证据可信度；
- 生成最终推荐等级。

### 13.5 面试生成 Agent

**输入：** 候选人短板 + 岗位要求  
**输出：** 面试问题和实操任务  

主要任务：

- 生成技术追问；
- 生成项目验证问题；
- 生成实操任务；
- 生成风险追问。

## 14. 核心算法规则

### 14.1 证据强度规则

```text
如果能力出现在项目经历中，并且有具体负责内容：
    标记为强证据

如果能力出现在项目经历中，但描述较笼统：
    标记为中证据

如果能力只出现在技能栏：
    标记为弱证据

如果岗位要求该能力，但简历中未出现：
    标记为缺证据
```

### 14.2 匹配评分规则

```text
技能匹配度 = 已匹配技能权重 / 岗位总技能权重

项目相关度 = 相关项目能力覆盖情况评分

证据可信度 = 强证据比例 × 100 + 中证据比例 × 70 + 弱证据比例 × 40

成长潜力 = 课程、竞赛、证书、学习经历综合评分

表达完整度 = 简历结构、项目描述完整度综合评分
```

### 14.3 推荐等级规则

```text
85 分以上：强烈推荐
75 - 84 分：建议进入一面
60 - 74 分：谨慎考虑
60 分以下：暂不推荐
```

## 20. 项目总结

**证聘 AI TrustHire AI** 是一款面向高校毕业生校招场景的可信招聘智能体。系统以“岗位能力图谱 + 简历证据链 + 人岗匹配评分 + AI 面试任务生成”为核心，解决传统招聘中简历筛选效率低、AI 简历包装严重、候选人能力难验证等问题。

该项目的核心创新不在于简单使用 AI 生成内容，而在于构建了一套面向招聘场景的可信判断机制，使岗位要求、简历内容、能力证据和面试验证形成闭环。

最终目标是：

> 帮助企业看到真实能力，帮助学生获得公平机会，让招聘从关键词匹配走向证据驱动匹配。
